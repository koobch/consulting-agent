# 자가발전 로그 (Self-Improvement Log)

> **이 파일은 리서치 시스템이 실행될 때마다 자동으로 업데이트된다.**
> 각 에이전트는 작업 완료 후 이 로그에 발견된 문제와 개선 사항을 기록한다.

## 목적
- 반복되는 실수를 패턴화하여 예방
- 에이전트별 품질 추이 추적
- 데이터 소스의 접근성 변화 기록
- 프레임워크/벤치마크의 유효성 검증

---

## 실수 패턴 DB (Error Pattern Database)

### EP-001: 채널/플랫폼 추측 오류
- **발생 빈도**: 높음
- **발생 에이전트**: data-researcher
- **패턴**: 특정 채널/플랫폼 중심 기업이라는 선입견으로 "단일 채널" 기재. 예: 모바일 중심 기업이라는 추측으로 "모바일 전용"이라 기재했으나 실제로는 PC/콘솔에도 진출
- **방지**: 각 제품별 domain-profile의 확인 소스에서 실제 유통 채널 직접 확인
- **자동 검증 트리거**: fact-checker Level 0에서 "~전용", "~단일" 서술이 있으면 채널/플랫폼 크로스체크 필수

### EP-002: 엔터티(그룹/자회사) 혼동
- **발생 빈도**: 높음
- **발생 에이전트**: data-researcher, financial-analyst
- **패턴**: 그룹 연결(CFS) 수치를 자회사(OFS) 수치로 잘못 사용
- **방지**: DART 조회 시 fs_div=OFS 필수, 모든 수치에 [그룹]/[별도] 라벨
- **자동 검증 트리거**: fact-checker가 재무 수치에 엔터티 라벨 없으면 FAIL

### EP-003: 제품/서비스 수 누락
- **발생 빈도**: 중간
- **발생 에이전트**: data-researcher
- **패턴**: 검색 결과에 노출된 주요 제품만 카운트, 마이너 제품/서비스 누락. 예: 기업 공식 사이트에 8개 제품이 있는데 뉴스에 노출된 5개만 기재
- **방지**: 기업명으로 domain-profile의 확인 소스에서 전체 제품/서비스 목록 확보
- **자동 검증 트리거**: fact-checker가 공식 홈페이지 제품 목록과 크로스체크

### EP-004: 회계연도/달력연도 혼동
- **발생 빈도**: 중간
- **발생 에이전트**: financial-analyst, data-researcher
- **패턴**: 기업의 회계연도(FY)를 달력연도(CY)와 혼동. 예: FY2025(=2024.4–2025.3 결산)를 CY2025 실적으로 오인
- **방지**: 해외 기업 재무 비교 시 FY/CY 구분 명시, 결산월 확인 필수

### EP-005: 사실 검증 없이 분석 시작
- **발생 빈도**: 치명적
- **발생 에이전트**: research-pm (워크플로우 단계 스킵)
- **패턴**: Ground Truth 없이 바로 가설 수립으로 진입
- **방지**: Step 0.5 팩트시트 → 사용자 승인 → 이후 단계 진행 (필수)
- **시스템 제약**: hypothesis-builder는 팩트시트 파일 없으면 작업 거부

### EP-006: 서술적 주장 미검증
- **발생 빈도**: 높음
- **발생 에이전트**: fact-checker (검증 누락)
- **패턴**: "라이선스 IP 100%", "특정 카테고리 중심" 등 텍스트 주장을 숫자만큼 엄격하게 검증하지 않음
- **방지**: Level 0 서술적 주장 검증을 최우선 실행
- **자동 검증 트리거**: "~100%", "~전용", "~중심", "~의존" 키워드 자동 탐지

### EP-007: SEC CIK 패딩 오류
- **발생 빈도**: 낮음 (시스템 에러)
- **발생 에이전트**: data-researcher, financial-analyst
- **패턴**: CIK를 패딩 없이 API URL에 사용하여 404
- **방지**: 항상 10자리 제로패딩 (CIK0000712515)

### EP-008: 시장 규모 전망치 혼용
- **발생 빈도**: 중간
- **발생 에이전트**: data-researcher
- **패턴**: 리서치 기관이 동일 연도에 대해 연중 여러 차례 전망치를 수정하여, "전망(forecast)"과 "실적(actual)"이 혼재됨. 예: 2024년 시장 규모 — $187.7B(2024 중반 전망) vs $182.7B(2025.06 실적 확인)
- **방지**: (1) 리서치 기관 수치 사용 시 반드시 "전망(E)" vs "실적(A)" 라벨 부착, (2) 가장 최신 발표 기준 실적을 우선 사용, (3) 전망치는 발표 시점 명시
- **자동 검증 트리거**: fact-checker가 리서치 기관 출처 시장 규모 수치에 전망/실적 라벨 없으면 WARNING

### EP-009: 비상장사 추정 매출 연도 혼입
- **발생 빈도**: 중간
- **발생 에이전트**: data-researcher
- **패턴**: 비상장사의 추정 매출(3rd-party 추정치)에서 연도 간 수치가 파일 간 교차 기재됨. 예: A사 2024 매출 $710M(정확)인데 다른 파일에서 $505M(2025 추정)으로 잘못 기재
- **방지**: (1) 비상장사 추정 매출은 반드시 [연도+출처] 태그 필수, (2) 복수 파일에서 동일 기업 데이터 참조 시 교차 정합성 확인, (3) 개별 제품 합산 = 전체 합산인지 산술 검증
- **자동 검증 트리거**: fact-checker Level 4에서 동일 기업 매출이 파일 간 10% 이상 차이 나면 FAIL

### EP-010: 수익 모델별 매출 패턴 미구분
- **발생 빈도**: 치명적
- **발생 에이전트**: data-researcher, insight-synthesizer, report-writer
- **패턴**: 일회성 구매 제품의 매출을 구독/지속 과금 제품처럼 연간 지속 매출로 추정. 일회성 구매 모델은 출시 후 1–3개월에 매출 70–80%가 집중되고 이후 급감하는 Front-loaded 패턴. 예: B2P 패키지 제품($39.99)을 "연 300–350억 지속 매출"로 잘못 추정
- **방지**: (1) 팩트시트에 각 제품의 수익 모델(domain-profile 참조) 필수 기재, (2) 일회성 구매 제품은 Front-loaded 매출 커브 적용 (출시 연도 피크 → 다음해 30–40% → 이후 부가 상품 의존), (3) 매출 전망 시 수익 모델별 템플릿 사용
- **자동 검증 트리거**: fact-checker가 일회성 구매 제품에 연 단위 지속 매출 전망이 있으면 WARNING + 매출 커브 확인 요구

### EP-011: 제품 출시일/종료일 미검증
- **발생 빈도**: 높음
- **발생 에이전트**: data-researcher
- **패턴**: 제품의 정확한 출시일·서비스 종료일을 확인하지 않고 "현재 운영 중"으로 기재. 예: 미출시 제품을 현재 운영 제품으로, 이미 종료된 서비스를 운영 중으로 기재
- **방지**: (1) 팩트시트에 각 제품의 정확한 출시일 + 종료일(해당 시) 필수 기재, (2) domain-profile의 확인 소스/공식 공지에서 출시일 직접 확인, (3) "서비스 종료" / "단종" / "EOL" 뉴스 검색 필수
- **자동 검증 트리거**: fact-checker가 팩트시트의 모든 제품에 대해 출시일·현재상태를 domain-profile의 확인 소스로 크로스체크. 출시일 누락 시 FAIL

### EP-012: 유통/개발 주체 간 매출 인식 구조 미반영
- **발생 빈도**: 높음
- **발생 에이전트**: data-researcher, financial-analyst
- **패턴**: 제품의 총매출(Gross Revenue)을 개발·생산 주체의 별도 매출로 혼동. 유통 주체-개발·생산 주체 관계에서 개발·생산 주체는 매출의 일부만 인식(라이선스/서비스 수수료). 예: 스토어 총매출 $40M을 개발사 별도 매출로 그대로 사용 → 실제 인식 매출은 25–31% 수준
- **방지**: (1) 팩트시트에 유통 주체-개발·생산 주체 관계 명시, (2) 개발·생산 주체 별도 재무제표의 매출을 기준으로 사용, (3) 총매출 추정치와 별도 매출의 배분율 산출, (4) 비교 시 동일 기준(총매출 vs 인식 매출) 사용
- **자동 검증 트리거**: fact-checker가 개별 제품 매출 합계와 별도 매출 총액의 정합성 확인. 차이 30% 이상 시 FAIL + 배분 구조 확인 요구

### EP-013: 전략 제안이 기업 내부 역량에 근거하지 않음
- **발생 빈도**: 치명적
- **발생 에이전트**: hypothesis-builder, insight-synthesizer, report-writer
- **패턴**: 전략 제안(신규 제품, 카테고리 확장, 채널/플랫폼 전환 등)을 시장 데이터와 경쟁사 트렌드만으로 도출하고, 해당 기업의 실제 역량 이력·카테고리 전문성·핵심 기술·성공/실패 패턴을 참고하지 않음. 예: 특정 카테고리 전문 기업에 경험 전무한 신규 카테고리 진출을 제안
- **방지**: (1) 팩트시트에 **역량 이력(Capability History)** 섹션 필수 포함 — 과거 전체 제품, 카테고리별 경험 횟수, 성공/실패 기록, 핵심 기술, (2) 가설 수립 시 "이 기업이 실제로 할 수 있는가?"를 검증 질문에 포함, (3) 인사이트의 전략 제안에 **역량 실현 가능성(Capability Feasibility)** 평가 필수, (4) 보고서의 신규 제품 제안은 기업 역량 이력과의 연결고리 명시
- **자동 검증 트리거**: fact-checker가 전략 제안/신규 제품에 대해 "이 기업이 유사 카테고리/채널/기술을 과거에 수행한 경험이 있는가?" 확인. 경험 없으면 WARNING + 리스크/전제조건 명시 요구

### EP-014: 과거 역량의 현재 적용 가능성 미검증
- **발생 빈도**: 높음
- **발생 에이전트**: data-researcher, insight-synthesizer, report-writer
- **패턴**: 10–20년 이상 전 역량(전신 회사 시절, 사명 변경 전)을 현재 역량으로 그대로 인용. 인력·기술·프로세스가 완전히 교체된 상태인데도 "전신 회사 시절 경험 → 현재 역량"처럼 연결
- **방지**: (1) 역량 이력에서 **현재 법인 기준 10년 이내** 제품만 "현재 역량"으로 분류, (2) 전신 회사 시절 제품은 **"레거시/연혁"**으로 분리하고 현재 역량과 명확히 구분, (3) 인력 연속성 증거(핵심 인력 잔류 여부) 없으면 과거 역량을 현재에 적용 금지
- **자동 검증 트리거**: fact-checker가 역량 분석에서 10년 이상 전 제품을 "현재 역량 근거"로 인용하면 WARNING

### EP-015: 그룹 내 자회사 간 제품 귀속 오류
- **발생 빈도**: 높음
- **발생 에이전트**: data-researcher
- **패턴**: 대형 그룹 산하 자회사 간 제품을 혼동. 유통 주체(그룹)가 동일하여 외부 표기로는 구분 불가. 예: 자회사B가 개발한 제품을 자회사A 개발로 잘못 기재
- **방지**: (1) 개발·생산 주체 확인 시 유통 주체가 아닌 **실제 개발·생산 조직** 확인, (2) 공시(DART 사업보고서 등)의 "주요 제품" 섹션에서 해당 법인이 직접 기재한 제품 목록만 사용, (3) 뉴스/보도자료에서 "개발·생산 주체" 표기 크로스체크
- **자동 검증 트리거**: fact-checker가 제품의 개발·생산 주체를 최소 2개 독립 소스로 확인. 유통 주체=개발·생산 주체로 단순 기재 시 FAIL

### EP-016: 시장 축소/하향 트렌드 리스크 미반영
- **발생 빈도**: 치명적
- **발생 에이전트**: insight-synthesizer, report-writer, hypothesis-builder
- **패턴**: 성장 전략에서 목표 카테고리/시장의 하향 트렌드를 검토하지 않음. 특히 기업의 핵심 역량 카테고리가 축소 중일 때 "핵심 역량이니까 계속 하자"는 논리만 제시하고, 축소 시장에서의 성장이 가능한지 비판적 검토 누락
- **방지**: (1) 전략 제안의 목표 카테고리에 대해 **시장 규모 추이(3년+)와 성장률** 필수 확인, (2) 축소 시장에 진입하는 전략은 "왜 축소 시장에서도 이 기업이 성장할 수 있는가?"를 명시적으로 답해야 함, (3) 대체 카테고리/시장 대비 기회비용 분석 포함
- **자동 검증 트리거**: fact-checker가 전략 제안의 목표 카테고리에 대해 최근 3년 시장 성장률 확인. 마이너스 성장 또는 전체 시장 대비 상대적 축소 시 WARNING + 리스크 명시 요구

### EP-017: 서술적 수치와 테이블 수치 불일치
- **발생 빈도**: 높음
- **발생 에이전트**: report-writer, insight-synthesizer
- **패턴**: 본문 서술에 기재한 요약 수치(예: "연 15–20% 감소")가 동일 보고서 내 상세 테이블의 실제 계산 값(18–26%)과 불일치. 서술 작성 시 테이블을 역산하지 않고 직감적으로 기재하여 발생
- **방지**: (1) 서술적 수치는 반드시 테이블 값에서 역산하여 기재, (2) 범위 표현 시 min/max를 테이블에서 직접 계산, (3) "약" 표현이라도 실제 값과 ±2%p 이내에서만 허용
- **자동 검증 트리거**: fact-checker가 본문 서술의 % 수치와 동일 보고서 내 테이블 수치를 교차 비교. 불일치 시 FAIL

### EP-018: 이전 수정 사항의 불완전 적용 (재위반)
- **발생 빈도**: 중간
- **발생 에이전트**: report-writer
- **패턴**: 한 번 정정된 사실이 보고서의 다른 섹션에서 재위반. 수정이 발생 지점에만 적용되고 보고서 전체에 걸쳐 전파되지 않음
- **방지**: (1) 사실 정정 시 보고서 전체에서 해당 키워드를 검색(grep)하여 모든 인용 지점을 수정, (2) 정정 체크리스트에 "전파 확인" 항목 추가
- **자동 검증 트리거**: fact-checker가 정정 사항에 대해 보고서 전문 검색 수행. 동일 오류가 다른 위치에 남아있으면 FAIL

### EP-020: 출처 표기 누락
- **발생 빈도**: 높음
- **발생 에이전트**: report-writer
- **패턴**: 보고서 본문에 수치/근거가 있지만 `[S##]` 출처 ID가 미표기. report-writer.md에 인라인 출처 규칙이 정의되어 있으나, 보고서 작성 시 출처 인덱스를 참조하지 않고 작성
- **방지**: (1) mechanical-validator 검증 10에서 수치 라인의 `[S##]` 유무 기계 검증, (2) report-auditor 영역 8에서 출처 완전성 감사, (3) report-writer에 부록 출처 인덱스 생성 의무 명시
- **자동 검증 트리거**: mechanical-validator가 숫자+단위 포함 라인에 `[S##]` 패턴 없으면 WARNING

### EP-022: 개별 제품 성공을 시장 성장으로 일반화
- **발생 빈도**: 치명적
- **발생 에이전트**: insight-synthesizer, report-writer
- **패턴**: 특정 제품의 히트를 해당 카테고리 "시장이 성장하고 있다"는 근거로 사용. 그러나 제품 성공은 IP·품질·마케팅 등 개별 요인의 결과이며, 카테고리 시장 추세와는 독립적. 예: 제품A 대성공($70M 첫달)을 "카테고리X 시장 성장 시그널"로 주장했으나 실제 카테고리 시장 데이터 없음
- **방지**: (1) 전략 제안의 **모든 목표 카테고리**에 대해 카테고리별 시장 성장률 데이터 필수 확보, (2) 개별 제품 성공 사례만으로 카테고리 추세를 주장하면 FAIL, (3) 시장 데이터 미확보 카테고리는 "시장 성장 미검증 — IP/품질 기반 접근"으로 명시
- **자동 검증 트리거**: fact-checker가 전략 제안에서 "시장 시그널", "성장 카테고리", "트렌드" 등의 표현이 있을 때 해당 카테고리의 시장 데이터(성장률/규모 추이) 존재 여부를 확인. 데이터 미확보 시 WARNING
- **관련 EP**: EP-016(시장 축소 리스크)의 반대 방향 — EP-016은 "축소 시장 미반영", EP-022는 "성장 미검증을 성장으로 단정"

### EP-021: 틸드(~) 문자의 GitHub 취소선 렌더링
- **발생 빈도**: 높음
- **발생 에이전트**: report-writer
- **패턴**: 범위 표현(600~800)이나 근사값(~550억)에 사용된 `~` 문자가 GitHub Flavored Markdown에서 `~~취소선~~`으로 렌더링됨
- **방지**: (1) 범위 표현은 en-dash(–)로 대체 (600–800), (2) 근사값은 `약`으로 대체 (약 550억), (3) mechanical-validator 검증 11에서 `~` 문자 자동 감지
- **자동 검증 트리거**: mechanical-validator가 코드블록 외부에서 `~` 문자 발견 시 WARNING

### EP-023: 출처-데이터 추적성 미검증 (팬텀 데이터)
- **발생 빈도**: 높음
- **발생 에이전트**: report-writer, data-researcher
- **패턴**: [S##] 출처 태그가 형식적으로 존재하지만, 해당 소스 파일에 인용된 수치가 실제로 없는 **팬텀 데이터** 문제. 에이전트가 리서치 과정에서 웹에서 본 수치를 소스 파일에 기록하지 않고 보고서에 직접 기재하면서 발생. 형식 검증(L1: [S##] 존재 여부)만으로는 탐지 불가
- **방지**: (1) `source-traceability-checker.py`로 보고서 작성 후 자동 검증 (unverified 0건 필수), (2) data-researcher에 소스 파일 완전성 규칙 추가 — 수집한 모든 수치를 소스 파일에 기록, (3) report-writer에 추적성 검증 절차 추가 — 수치 기재 시 소스 파일에서 직접 확인, (4) fact-checker에 보고서 사후 검증(Post-Report Verification) 단계 추가
- **자동 검증 트리거**: fact-checker가 보고서 완성 후 `source-traceability-checker.py` 실행. unverified 클레임 1건 이상이면 FAIL

### EP-019: KPI/목표치와 산출 근거 간 논리 갭
- **발생 빈도**: 중간
- **발생 에이전트**: report-writer, financial-analyst
- **패턴**: KPI 테이블의 목표치(예: 자체 IP 매출 비중 48%)가 본문의 제품별 매출 합산으로 역산 시 달성 불가능하거나, 미설명 기여분(M&A 등)이 암묵적으로 포함됨
- **방지**: (1) KPI 목표치는 반드시 본문 내 산출 근거와 수식적으로 연결, (2) 외부 기여(M&A 등)가 포함된 경우 명시적으로 기재, (3) 역산 검증: KPI % × 전체 매출 = 해당 항목 합계인지 확인
- **자동 검증 트리거**: fact-checker가 KPI 테이블의 % 목표를 절대 금액으로 변환한 뒤, 본문 제품별 합계와 비교. 20% 이상 괴리 시 FAIL + 산출 근거 명시 요구

### EP-024: 시나리오 분석 시 보수적 base case 누락 + 정렬 오류
- **발생 빈도**: 중간
- **발생 에이전트**: insight-synthesizer, report-writer
- **패턴**: 외부 파트너/계약에 의존하는 시나리오 분석에서 "모든 외부 요인이 실패하는 경우" 시나리오가 누락됨. 또한 시나리오를 알파벳순으로 나열하여 의사결정자가 가장 중요한 시나리오를 식별하기 어려움
- **방지**: (1) 시나리오 분석 시작 전 "모든 외부 의존 요인이 실패하는 경우" 체크 — 해당 시나리오를 안전망(fallback)으로 반드시 포함, (2) 시나리오 테이블은 추천도/중요도 순서로 정렬 — 알파벳순 금지, (3) 안전망 시나리오는 발생 확률과 무관하게 명시

## 로그 기록 규칙

### 누가 기록하는가
- **data-researcher**: EP 패턴 발생 시, 데이터 소스 접근성 변화 시
- **fact-checker**: 검증 실패 패턴 발견 시, 새로운 EP 추가 시
- **research-pm**: 프로젝트 완료 후 에이전트 품질 추이, 프레임워크 적용 기록
- **financial-analyst**: 벤치마크 갭 발견 시

### 언제 기록하는가
- **즉시 기록**: EP 패턴 발생 시, 데이터 소스 차단 발견 시
- **배치 후 기록**: 각 배치 검증 완료 후
- **프로젝트 완료 후 기록**: 전체 품질 추이, 프레임워크 효과 평가

### 어떻게 활용하는가
1. **research-pm**: 다음 리서치 시작 시 이 로그를 읽고, EP 패턴을 해당 에이전트에게 사전 경고
2. **fact-checker**: EP 패턴 DB를 검증 체크리스트에 자동 포함
3. **data-researcher**: 데이터 소스 접근성 로그를 보고 차단된 소스 회피
4. **framework-designer**: 프레임워크 적용 기록을 참조하여 케이스별 추천 정확도 향상
